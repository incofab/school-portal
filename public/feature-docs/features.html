How To Use This File

This inventory is the project-level feature map. Before changing behavior, locate the feature here, then inspect the referenced route files, controllers, actions, models, React pages, and tests. The implementation is a Laravel backend with Inertia + React/TypeScript frontend, Chakra UI components, Pest feature tests, and feature-specific route files included from routes/institution.php.

Maintenance rule: every feature addition, behavioral change, new route, new workflow, data model change, or UI surface change must update public/feature-docs/. If a feature changes, update this inventory and any affected backend.html, frontend.html, data.html, recommendation, or proposal page.

Shared Frontend Shell

The Inertia React app is bootstrapped in resources/js/app.tsx. It resolves pages from resources/js/pages/**/*.tsx, mounts a single React 18 root, and wraps page output with React StrictMode, ChakraProvider, ColorModeScript, and ProSidebarProvider. The shell renders pages directly inside those providers without an artificial first-load spinner or delayed loading state. Institution dashboard pages also expose a route-aware Guide button in the shared dashboard header; it opens a modal explanation for the current critical feature area using resources/js/components/explanations/dashboard-explanations.ts and resources/js/components/explanations/dashboard-explanation-guide.tsx.

Implemented Features

00 Shared Media Tracking and Upload Lifecycle

All first-party uploads that end up on the public S3 disk now pass through a shared media-management layer that creates a database record before upload, stores file metadata and ownership details, and then marks the media row uploaded or failed after transfer. Legacy URL/path columns are still synchronized for compatibility while the new media table becomes the auditable source of truth.

  • Backend: App\Support\Media\MediaManager, App\Models\Media, App\Traits\HasMedia, and media:migrate-legacy.
  • Migrated upload flows: institution profile photos, institution setting images, user avatars, institution-group banners, admission application photos, manual payment proofs, TinyMCE/CCD inline images, and imported CCD course-content image assets.
  • Migration: php artisan media:migrate-legacy backfills media records from legacy model fields and can scan the S3 disk for unattached legacy files.

14 Result Recording, Processing, Locking, Publishing, and Reports

Results are a major domain. The app records individual course results, bulk uploads course result sheets, records whole-class sheets, downloads result recording sheets, lists and deletes course result info summaries, calculates class result info, recalculates aggregate results, generates term results, generates session results, computes grades and positions, supports subject reports, dedicated grade report sheets with overall term-result distribution plus subject-by-grade matrices, cumulative result sheets, term details, teacher/principal comments, AI-generated teacher/principal comments from class result analysis, extra result data, class result sheet rendering, transcripts, signed result sheets, public result activation, result publications, and result delivery to guardians. WhatsApp check-result requests now reuse shared result access logic for current institution settings, result publication, PIN activation, recent-result fallback selection, and signed result URL behavior before sending one or more result links back through the Meta WhatsApp Cloud API webhook. The public signed result-sheet route enforces Laravel signed URL validation. Score recording/updating/deletion, result processing start/completion/failure, lock/unlock, and publication workflows now emit explicit integrity audit events with class, course, student, teacher, session, term, count, and old/new score metadata. Result publication reads a fresh copy of the institution's current session and term settings before billing/publishing so repeated publish requests in the same process use the latest settings. Platform admins can set a fixed partner commission on result-checking price lists; result publication uses it instead of the partner percentage and credits any referring partner as a percentage of that fixed partner commission. Term details can override the institution default for whether exam scores appear on result templates; when unset, the institution result setting controls templates 1 through 8. Institutions can preview their selected result template from result settings; the preview renders a non-persisted dummy result sheet with 10 to 14 subjects, using existing institution courses where available and fallback subjects when needed.

Course result upserts are protected by lookup constraints on student/course/class/session/term/mid-term keys, preserve existing exam or assessment values during partial updates, honor assessment term/mid-term/class filters and dependency mappings, and class/course summary lookups use indexed keys to reduce lock contention during result recording and exam result transfer. Course and class recording pages now receive one term-specific assessment group payload containing full-term and mid-term lists, switch the visible fields based on the selected result mode, visually distinguish mid-term from full-term entry, and hide the exam input according to the institution result exam mode setting. When mid-term results are enabled, the class recording page hides student score fields until staff explicitly choose full-term or mid-term recording, and the course result upload modal hides the Excel upload field and save action until staff explicitly choose full-term or mid-term upload. The single student course result recording page keeps result type, session, term, and student selection in the URL; when a student is selected it preloads the matching existing result for that key so teachers edit the current row instead of recording blindly, and while those selections are reloading the page it locks the entry controls and action buttons with a visible loading notice so unsaved edits are not changed during the refresh. A student-wide subject recording page lets staff select one student, then record that student's results across all eligible class subject mappings in one submission while reusing the same assessment, exam visibility, lock, grading, audit, and course-summary processing rules. The student-wide and class recording pages submit only new or changed score rows, avoiding unchanged prefilled results in the payload.

Class result info records now have an is_locked state. Calculation and recalculation lock the result info; authorized users or form teachers can unlock/relock; recording, updating, deleting course results, deleting course result info summaries, and bulk processing course/term results checks the lock before mutating result data.

  • Routes/controllers: course-results and class-result-info routes in routes/institution.php, CourseResultsController, RecordClassResultController, RecordStudentSubjectsResultController, ClassResultInfoController, ListCourseResultInfoController, TransferCourseResultInfoController, ListTermResultController, DeleteTermResultController, DummyResultSheetController, ResultPublicationsController, SubjectReportController, SingleSubjectReportController, GradeReportController, student result controllers.

17B Recruitment and Vacancy Applications

Institutions can create and publish vacancy posts for public recruitment, advertise vacancies on an institution-scoped public board, and collect public applications without file uploads. Applicants can provide CV, cover-letter, and portfolio links, must type or link a cover letter, and receive an application number after submission. Institution admins review applications from a specific vacancy post, open applicant links, store review notes, and move candidates through pending, shortlisted, hired, and declined states. The application model is intentionally institution-scoped and candidate-centered so it can later be attached to CBT screening flows in the same way admission applications are used for admissions exams. Recruitment applicants can now enter an event code and their recruitment application number on the public recruitment exam login page to create or resume a started CBT exam tied directly to the recruitment application record.

28 Audit Logs and Activity Monitoring

A reusable audit logging foundation records activity events with actor, subject, institution, institution group, request context, severity, retention category, JSON properties, old/new value snapshots, and optional row-hash integrity metadata. The logger is safe for HTTP requests, jobs, commands, tests, and guest/system events, and centrally redacts sensitive keys before persistence. Admin-manager and institution admin list pages expose paginated, searchable, filterable activity logs with advanced filters for date range, institution, institution group, actor, actor role, category, event, subject type/name, severity, IP address, request id, retention category, and impersonated-only activity. Admin managers can export the global filtered scope, while institution admins can export only the current institution scope. Manager users must enter institution pages through impersonation; legacy activity-log permissions do not grant direct institution-page access. The detail drawer shows actor, subject, institution/group, event/category/severity, request and impersonator context, properties, old/new values, a compact diff, integrity hash, and financial summary fields for amount/reference/status/provider/account metadata. Automatic model auditing now records opted-in high-value create, update, delete, restore, and force-delete events without controller changes. Domain workflows suppress same-write automatic model rows when a richer explicit event is emitted, avoiding duplicate activity log entries. SecurityActivityLogger records explicit authentication, authorization, account, role, and impersonation domain events, AcademicActivityLogger records explicit student, guardian, course, curriculum, assignment, and attendance workflow events, AcademicIntegrityActivityLogger records assessment/result/exam/admission integrity events, and FinancialActivityLogger records explicit finance, wallet, payroll, expense, bank-account, commission, and external payment integration events.

  • Backend: App\Support\Audit\ActivityLogger, SecurityActivityLogger, AcademicActivityLogger, AcademicIntegrityActivityLogger, FinancialActivityLogger, AuditRequestContext, ActivityLogSanitizer, ActivityLogQuery, ActivityLogExporter, ModelAuditRegistry, ModelAuditObserver, ActivityLogPolicy, manager/institution ActivityLogController, ActivityLogUITableFilters, audit:prune, and enum-backed ActivityLogCategory/ActivityLogSeverity filter options.
  • Routes: managers.activity-logs.index and managers.activity-logs.export for admin-manager viewing/export, plus institutions.activity-logs.index and institutions.activity-logs.export for current-institution admin viewing/export, including group-scoped logs for the institution's own institution group only.
  • Frontend: shared components/activity-logs/activity-log-list.tsx, manager managers/activity-logs/list-activity-logs.tsx, and institution institutions/activity-logs/list-activity-logs.tsx.
  • Automatic coverage: institutions/groups/onboarding requests/partners/users/students/guardians/classes/courses/curriculum/e-library/assignments/attendance, assessments/learning evaluations, result records, exam/event records, payment references/receipts/fundings/transactions/reserved accounts, admissions/application records, vacancy posts, timetable/school-activity/live-class records, association records, payroll adjustments/types, and Media. Fee/manual-payment/withdrawal/bank-account/commission/payroll/expense workflows prefer explicit finance events to avoid duplicate generic logs.
