How To Use This File

This inventory is the project-level feature map. Before changing behavior, locate the feature here, then inspect the referenced route files, controllers, actions, models, React pages, and tests. The implementation is a Laravel backend with Inertia + React/TypeScript frontend, Chakra UI components, Pest feature tests, and feature-specific route files included from routes/institution.php.

Maintenance rule: every feature addition, behavioral change, new route, new workflow, data model change, or UI surface change must update public/feature-docs/. If a feature changes, update this inventory and any affected backend.html, frontend.html, data.html, recommendation, or proposal page.

Primary User Surfaces

Public and Guest

Home, institution registration, partner registration, login, password reset, student login, admission application, admission form purchase, admission letters, public result activation, public exam result lookup, and external exam entry points. Start in routes/web.php.

Institution Staff

Institution dashboard, setup checklist, profile/settings, users, students, guardians, classes, courses, assessments, results, exams, attendance, payments, payroll, messaging, notifications, curriculum, activities, and timetable operations. Start in routes/institution.php.

Students and Guardians

Student login, guardian dependents, impersonation, result sheets, transcripts, student fee payments and receipts, exam pages, and admission/exam access. Related controllers are under app/Http/Controllers/Institutions/Students and .../Guardians.

Managers and Admins

Manager dashboard, institution groups, institutions, registration requests, partner requests, managers, admin user listing and impersonation, academic sessions, platform funding, billings, commissions, bank accounts, withdrawals, and notifications. Start in routes/manager.php.

Implemented Features

01 Authentication, Account Access, and Impersonation

Supports staff login, student login, password reset, authenticated dashboard routing, user password changes, BVN/NIN updates, logout, manager-only impersonation of users/institutions, admin user-list impersonation, and stopping impersonation. Public result activation and exam-result login are also exposed to guest users.

  • Routes: routes/web.php, routes/api.php.
  • Controllers: AuthController, StudentAuthController, PublicExamResultController, TermResultActivationController, Users/*, Impersonate/*.
  • Frontend: resources/js/pages/auth/*, resources/js/pages/users/*.
  • Middleware: Authenticate, VerifyManager, VerifyInstitutionUser, VerifyAdmin, StudentAccess.
  • Data: User, Admin, InstitutionUser, Student, Spatie permission tables.
  • Tests: UserControllerTest, TermResultActivationControllerTest.

02 Institution and Partner Onboarding

Schools can submit registration requests after reading the public institution value and authorised-representative notice, partners can submit partner registration requests after reading the public partner opportunity and commission message, and managers can convert approved requests into institution groups, institutions, and partner records. Public admission routes also support institution-scoped form search and purchase flows.

  • Routes: public register/{partner?}, partner-registration, manager registration-requests, partner-registrations, institutions/create.
  • Controllers: InstitutionRegistrationRequestController, PartnerRegistrationRequestController, Managers/RegistrationRequests/RegistrationRequestsController, Managers/PartnerRequests/PartnerRegistrationRequestsController, Managers/Institutions/InstitutionRegistrationController.
  • Actions and models: RegisterInstitution, RegisterInstitutionGroup, RecordPartner, RegistrationRequest, PartnerRegistrationRequest, Partner.
  • Frontend: auth/register.tsx, auth/register-partner.tsx, auth/registration-form.tsx, manager registration request pages.
  • Tests: institution and partner registration request controller tests.

03 Manager Administration, Institution Groups, and Academic Sessions

Managers administer platform-level entities: institutions, institution groups, managers, user lookup, billing, commissions, fundings, bank accounts, withdrawals, notifications, and academic sessions. Admin managers can list platform users with search and role/institution/status filters, see each manager wallet balance, then impersonate any user from the list. The manager dashboard now surfaces pending withdrawal requests in its attention notice section for admins, and the manager withdrawals list shows the accountable requester name for each withdrawal. Institution groups include branding, banner/background, domain, wallet balance, loan limit, status, brand color, and invoice generation by academic session and term.

  • Routes: routes/manager.php, especially users, institution-groups, institutions, academic-sessions, funding, billings, commissions, withdrawals.
  • Controllers: Managers/Users/ListUsersController, Managers/InstitutionGroups/InstitutionGroupsController, Managers/Institutions/*, Managers/AcademicSessions/AcademicSessionController, Managers/ManagerController.
  • Frontend: resources/js/pages/managers/*, including academic-sessions/list-academic-sessions.tsx and academic-sessions/create-edit-academic-session.tsx.
  • Data: InstitutionGroup, Institution, AcademicSession, Funding, Commission, Withdrawal, BankAccount.
  • Tests: manager funding, institution registration, registration request, and academic session tests.

04 Institution Dashboard, Profile, Settings, and Setup Checklist

Institution users get a dashboard, setup checklist, profile editing, photo upload, payment key settings, result settings, current academic session and term settings, and institution-level search endpoints. Admin and accountant dashboards now include a top attention section showing unread chats, the current pending manual-payment count with a quick link to review requests, plus a bank-account configuration prompt when the institution group has not added any bank accounts yet. Shared props expose current institution, session, term, and role context to the React app.

  • Routes/controllers: InstitutionController, InstitutionSettingController, SettingsHandler, SetupChecklistHandler, InstitutionDashboardStat.
  • Frontend: institutions/dashboard.tsx, institution-profile.tsx, settings/create-edit-institution-settings.tsx, settings/result-settings.tsx, settings/payment-keys-settings.tsx.
  • Data: Institution, InstitutionSetting, TermDetail, AcademicSession.

05 Users, Staff, Roles, and Staff Assets

Institution staff can create users, list/search users, update profiles, upload photos, bulk upload staff, download staff recording templates, update institution-user status, reset passwords, change roles, delete users, assign course-teacher records, and print staff ID cards.

  • Data: User, Staff, InstitutionUser, CourseTeacher, roles/permissions.

06 Student Records and Lifecycle

Student management covers CRUD, class-scoped student tiles, student code update, bulk upload, recording template download, student ID cards, fee-payment views, transcripts, term result details, session result sheets, class changes, multiple student class changes, migration, promotion, movement history, and movement rollback.

  • Data/tests: Student, StudentClassMovement, SessionResult; student management, promotion, class change, and revert tests.

07 Guardians and Dependents

Guardian workflows include class-based guardian recording, guardian listing, assigning students to guardians, guardian dependent listing, guardian impersonation into a student account, and dependent removal.

08 Classifications, Groups, Divisions, and Class Mapping

The class structure uses classifications for classes/arms, classification groups for broader promotion/result grouping, class divisions for cross-class groupings, and mappings/classifiable links. It supports search, bulk class creation, download/upload, student status updates per classification, class migration, promotion, division membership management, and per-class shortcuts to filtered lesson plans and topics.

  • Controllers: ClassificationController, ClassificationGroupController, ClassDivisionController, DownloadClassStudentsController, promotion and movement controllers.
  • Frontend: classification, classification group, and class division pages under resources/js/pages/institutions.
  • Data: Classification, ClassificationGroup, ClassDivision, StudentClassMovement, class_division_mappings, classifiable.
  • Tests: classification, class division, promotion, update class, and revert movement tests.

  • Controllers/actions: CoursesController, CCD/*, UploadCourseContent, ExportCourse, Questions/ExtractDocumentContent, Questions/ConvertDocumentToQuestions, GoogleAiHelper.

Curriculum features include institution topics and sub-topics, searchable/filterable scheme-of-work lists, searchable/filterable lesson-plan lists, lesson notes with class/subject/teacher/status/term filters, AI lesson-note generation, publication toggles, and detailed show/list/edit pages. The classes, subjects, and subject-teacher lists link each row directly to filtered lesson plans and topics, and the classes list also links each class to filtered lesson notes.

  • Controllers: Curriculums/TopicController, SchemeOfWorkController, LessonPlanController, LessonNoteController.

14 Result Recording, Processing, Locking, Publishing, and Reports

Results are a major domain. The app records individual course results, bulk uploads course result sheets, records whole-class sheets, downloads result recording sheets, lists and deletes course result info summaries, calculates class result info, recalculates aggregate results, generates term results, generates session results, computes grades and positions, supports subject reports, grade report summaries for term results, cumulative result sheets, term details, teacher/principal comments, extra result data, class result sheet rendering, transcripts, signed result sheets, public result activation, result publications, and result delivery to guardians.

Course result upserts are protected by lookup constraints on student/course/class/session/term/mid-term keys, preserve existing exam or assessment values during partial updates, honor assessment term/mid-term/class filters and dependency mappings, and class/course summary lookups use indexed keys to reduce lock contention during result recording and exam result transfer.

Class result info records now have an is_locked state. Calculation and recalculation lock the result info; authorized users or form teachers can unlock/relock; recording, updating, deleting course results, deleting course result info summaries, and bulk processing course/term results checks the lock before mutating result data.

  • Routes/controllers: course-results and class-result-info routes in routes/institution.php, CourseResultsController, RecordClassResultController, ClassResultInfoController, ListCourseResultInfoController, TransferCourseResultInfoController, ListTermResultController, DeleteTermResultController, ResultPublicationsController, SubjectReportController, student result controllers.
  • Actions: RecordCourseResult, RecordClassSheet, ProcessTermResult, ProcessSessionResult, ProcessClassGroupInfo, ClassResultInfoAction, GetGrade, GenerateSubjectReport, GenerateCourseSessionResult, TermResultDeleteHandler, DownloadResultRecordingSheet, DownloadCourseResult, SendTermResultToGuardians.
  • Frontend: institutions/courses/list-course-results.tsx, record-course-result.tsx, record-class-course-result.tsx, list-class-result-info.tsx, list-course-result-info.tsx, upload-class-sheet.tsx, result-sheets/*, session-result-sheets/*, results/list-term-results.tsx, result-publications/*, transcript pages.
  • Data/tests: CourseResult, CourseResultInfo, ClassResultInfo, ClassGroupResultInfo, TermResult, SessionResult, ResultPublication, TermDetail; broad result action and controller tests including lock behavior.

16 Exams, Events, CBT, External Exams, and Offline Mock API

Exam features include events, event courseables, exams, exam courseables, event content download, result transfer from events into term/course results, multi-transfer, student/admission external exam login, token-user profile update, event display, external exam creation, objective/theory question switching within each exam subject, manual theory answer evaluation by authorized staff, exam display/pause/end/result pages, leaderboards, public exam result lookup, and offline mock API endpoints for client testing and offline scenarios.

Event result transfer records all matching student scores inside a short retried transaction, then recalculates course result info and positions once after the transaction commits.

  • Routes: routes/exam.php, exam sections in routes/web.php, offline mock section in routes/api.php.
  • Controllers/support: Institutions/Exams/*, Institutions/Exams/ExamPage/*, Institutions/Exams/External/*, API/OfflineMock/*, CreateExam, EventResultHandler, ExamHandler, MockEventHandler, MockExamHandler, ExamAttemptFileHandler.
  • Frontend: institutions/exams/*, public/examiner, auth/student-exam-login.tsx, auth/admissions-exam-login.tsx, auth/exam-result-login.tsx.
  • Data/tests: Event, EventCourseable, Exam, ExamCourseable with objective score/count, theory awarded/max score, theory question count, per-question theory score map, and theory evaluation flags, TheoryQuestion, TokenUser; exam, transfer, external exam, offline mock, exam handler, and file handler tests.
